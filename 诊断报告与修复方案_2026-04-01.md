## 一、诊断现有问题

### 1.1 主题渲染问题（🔴 严重）

**现象：** 暗色主题"一半黑一半白"，样式未完全应用

**根本原因：**

1. **VitePress 1.x 不支持 `<script setup>` 语法**
   - 项目详情页（如 `projects/memory-boost.md`）使用了：
   ```vue
   <script setup>
   const stars = 0
   </script>
   ```
   - 这导致 Vue 模板语法未编译，直接显示源码 `{{ stars }} ⭐`

2. **CSS 选择器优先级问题**
   - `custom.css` 中定义了 `:root` 和 `html.dark`，但部分 VitePress 内部组件使用了更高优先级的内联样式
   - 代码块（`<pre>`）使用了 `shiki-themes github-light github-dark`，依赖 VitePress 的 `vp-adaptive-theme` 机制

3. **主题切换按钮未完全隐藏**
   - 虽然在 CSS 中添加了 `.VPNavBarAppearance { display: none !important; }`
   - 但配置中 `appearance: false` 可能未在所有 locale 中生效

**受影响文件：**
- `projects/*.md`（所有项目详情页）
- `zh/projects/*.md`（所有中文版项目详情页）
- `.vitepress/theme/custom.css`
- `.vitepress/config.js`

---

## 二、实时下载量功能调研


**结论：** ❌ **ClawHub 没有公开 API**

---

### 2.2 网页抓取可行性

**方案：** 使用前端 JavaScript 抓取 ClawHub 页面

**问题：**
1. **跨域限制（CORS）**
   - ClawHub 服务器未设置 `Access-Control-Allow-Origin: *`
   - 浏览器会阻止前端直接请求

2. **反爬机制**
   - 频繁请求可能触发速率限制
   - Vercel 部署的静态站点不适合做爬虫

3. **HTML 结构变化**
   - 依赖 HTML 解析，ClawHub 页面结构变化会导致失效

**结论：** ❌ **不可行（跨域问题无法解决）**

---

### 2.3 替代方案

#### 方案 A：构建时预取（推荐）⭐

**原理：** 在 `npm run build` 时使用 Node.js 脚本抓取下载量

**优点：**
- ✅ 无跨域问题（Node.js 不受 CORS 限制）
- ✅ 数据准确（直接从 ClawHub 页面提取）
- ✅ 零运行时开销

**缺点：**
- ⚠️ 需要手动触发构建才能更新数据
- ⚠️ 需要解析 HTML（依赖页面结构稳定）

**实现步骤：**
1. 创建 `scripts/fetch-downloads.js`
2. 使用 `node-fetch` + `cheerio` 抓取 ClawHub 页面
3. 更新 `_data/downloads.json`
4. 提交并触发 Vercel 构建

---

#### 方案 B：手动更新（当前方案）

**原理：** 保持现状，定期手动更新 `_data/downloads.json`

**优点：**
- ✅ 简单可靠
- ✅ 无额外依赖
- ✅ 完全控制

**缺点：**
- ⚠️ 数据可能过时
- ⚠️ 需要人工介入

**建议频率：** 每周或每次发布新技能时更新

---

#### 方案 C：第三方服务（不推荐）

**原理：** 使用如 `shields.io` 等徽章服务

**问题：**
- ❌ ClawHub 不在支持列表中
- ❌ 需要 ClawHub 提供 API 才能集成

**结论：** ❌ **不可行**

---

### 2.4 推荐方案

**采用：方案 B（手动更新）+ 方案 A（未来优化）**

**理由：**
1. 当前下载量变化不频繁（每周检查一次足够）
2. 方案 A 需要额外开发时间，ROI 不高
3. 可以在未来下载量重要时再实施方案 A

**实施建议：**
- 在 `README.md` 中添加更新指南
- 设置每月 1 号提醒更新下载量
